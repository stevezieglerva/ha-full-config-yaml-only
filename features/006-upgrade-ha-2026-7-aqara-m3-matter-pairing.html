🏠 Home Assistant 2024.7 → 2026.7 Upgrade Plan

Fixing Aqara Hub M3 Matter pairing  •  Prepared July 5, 2026  •  Ziegler home

🔎 Root cause — confirmed

HA Core 2024.7.0 runs the Matter integration against python-matter-server 6.2.2 — a Matter 1.2-era stack from mid-2024. The Aqara M3 (firmware 4.3.7+) commissions as a Matter 1.3+ bridge, which that server cannot handle. The failure signature is the well-documented "Discovery timed out / Secure Pairing Failed" (core #134009). Matter itself works on this box — 9 devices / 71 entities (Eve, ThirdReality, Tapo) paired fine as simple end devices. This is a version problem, not a setup problem.

🎯 Target: 2026.7.x (latest patch)

The update dialog's "latest 2026.1.2" is a stale update entity — actual latest is 2026.7.1 (July 3, 2026). 2026.7 ships Matter Server 9.0, rewritten on matter.js with Matter 1.5.1 and dramatically better commissioning. Anything older still runs the end-of-life python-matter-server. 2026.7 is the release that makes the M3 pair.

Upgrade path — stepped, ≤6 releases per hop

Official guidance: HA keeps migration code for ~6 releases, so a 24-release jump must be stepped (docs). Update HAOS first (Settings → System → Updates), then hop Core:

  1. 2024.7.0today — backup + HAOS 18.x first
  2. 2025.1.xbackup encryption arrives — save the key!
  3. 2025.7.xcalendar add_event removed (not used ✓)
  4. 2026.1.xold HACS breaks — update HACS first
  5. 2026.7.xMatter Server 9.0 → pair the M3 🎉

At every hop: full backup → update HACS + custom components → update Core → let the recorder schema migration finish (long first boot on a 2-year-old SQLite DB — do not power-cycle) → clear the Repairs dashboard → next hop. Keep free disk ≥ 2× database size.

Gotchas checklist (in order)

  1. Full backup first. From 2025.1 onward backups are AES-encrypted by default — immediately save the emergency-kit encryption key or backups are unrecoverable. Automatic backups default to 04:45.
  2. Update HAOS 12.4 → 18.x before any Core hop (Supervisor stops offering updates on an OS older than the last 4 majors).
  3. HACS 2.0.5 → current before the 2026.1 hop — old HACS fails to load on 2026.1.2 (hacs #5044).
  4. Alexa Media Player 4.10.3 → ≥ 5.15.5 (current 5.15.6; 5.15.5 fixes an HA 2026.7 issue). A 4.x AMP will not survive the jump. Expect a full Amazon re-login with 2FA — the school wake-up/departure announcements depend on it; verify after upgrade.
  5. iCloud3 3.0.5.2 → ≥ 3.5.1. Apple changed authentication in early 2026; old versions can't sign in at all. Expect Apple re-auth/2FA.
  6. Replace cync_lights (breaks on 2025.3 color-mode enforcement) with the official Cync integration added in 2025.10 at the 2026.1 hop; audit automations for changed Cync entity IDs.
  7. Update the Zigbee2MQTT add-on to current 2.x before the final hop — 2026.4 ignores object_id in MQTT discovery, and Z2M 2.0 has its own breaking changes (read its changelog then).
  8. Recorder migrations: long first boots per hop on SQLite; never interrupt mid-migration.
  9. Post-landing audits: template binary sensors returning None now read unknown not off (2025.8); mobile-app trackers report zone friendly names (2025.11) — check tracker-based automations; nested variables scoping changed (2025.3/.4); Google Calendar omits declined events (2024.11) — relevant to school-calendar automations; HomeKit Bridge naming precedence changed (2025.5) — spot-check Apple Home.
  10. Matter Server 9.0 migration (2026.7): auto-migrates on first start, existing 9 Matter devices are not re-commissioned, no rollback exists. Backup + disable the app watchdog for the first start; power-cycle any device that shows offline (mDNS re-announce).
  11. 2026.7 automation editor defaults to purpose-specific triggers — UI-only change; classic YAML triggers unaffected.

Pairing the Aqara M3 (after landing on 2026.7)

  1. In the Aqara app, update the M3 firmware to ≥ v4.3.7.
  2. Aqara app → Hub M3 device page → "Third-Party Matter Ecosystems" → Matter Pairing Code. This per-ecosystem code (not the printed QR) is required once the hub is already in Aqara's fabric.
  3. HA → Settings → Devices & Services → Add Integration → Matter → "No, it's new" → More options → Matter Accessory → paste the code.
  4. Keep the M3 and HA on the same subnet/VLAN with mDNS and IPv6 link-local working.
  5. Bridged Zigbee children (sensors, switches, plugs, locks, curtains) appear via Matter. The IR blaster and alarm/siren do NOT come through. Fallback if Matter misbehaves: pair the M3 via the HomeKit Device integration (exposes more, incl. the security system).
  6. Leave existing Aqara devices on Zigbee2MQTT — moving any to the M3 changes device IDs and breaks the 187 device_id:-based automations.
